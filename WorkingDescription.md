# Концепция приложения для визуализации цепочек крафтов Minecraft

## Общая идея

Необходимо разработать десктопное приложение для построения интерактивных графов рецептов Minecraft.

Программа должна поддерживать как ванильный Minecraft, так и моды. Пользователь может загрузить один или несколько `.jar` файлов модов, после чего приложение автоматически извлекает из них предметы, рецепты, производственные блоки и связанные данные.

Главная задача приложения — помочь игроку визуально построить всю цепочку производства любого предмета в виде графа.

В качестве референсов можно использовать графы производства из Factorio, Satisfactory и подобные схемы зависимостей ресурсов.

---

# Основные возможности

## Импорт модов

Пользователь может:

* загрузить один или несколько `.jar` файлов;
* загрузить целый модпак;
* обновлять список модов без перезапуска проекта.

После загрузки приложение должно извлечь:

* все предметы;
* все блоки;
* все рецепты;
* все машины и станции крафта;
* дополнительные параметры рецептов.

Для каждого рецепта необходимо сохранять:

* входные ресурсы;
* выходные ресурсы;
* тип крафта;
* устройство, в котором выполняется крафт;
* время производства;
* вероятность получения результата (если есть);
* побочные продукты;
* данные мода-источника.

---

# Рабочая область

Основной интерфейс представляет собой бесконечный холст (canvas).

На нем пользователь строит граф производства.

Поддерживаются:

* масштабирование колесиком мыши;
* перемещение холста;
* выделение нескольких узлов;
* удаление узлов;
* автоматическое выравнивание графа;
* ручное расположение узлов.

---

# Добавление предмета

Пользователь выбирает предмет через поиск.

После выбора предмет появляется на холсте как целевой ресурс.

Например:

```
Алмазная кирка
```

или

```
Механическая рука (Create)
```

---

# Создание рецепта

Каждый предмет отображается как узел ресурса.

От любого входного ресурса пользователь может вытянуть соединительную линию.

После отпускания мыши открывается окно выбора рецепта.

Если рецептов несколько, пользователь видит список:

* верстак;
* печь;
* дробилка;
* механический пресс;
* рецепт из другого мода;
* альтернативный рецепт.

После выбора создается новый узел рецепта.

---

# Типы узлов

## Узел ресурса

Отображает:

* иконку предмета;
* название;
* количество.

Пример:

```
Железный слиток ×4
```

---

## Узел рецепта

Отображает:

* блок или машину;
* входные ресурсы слева;
* выходные ресурсы справа;
* время производства;
* скорость работы.

Структура:

```
[Входы] -> [Верстак] -> [Выходы]
```

или

```
[Входы] -> [Печь] -> [Выходы]
```

---

# Внешний вид узла рецепта

В центре:

* иконка устройства производства.

Слева:

* все входные ресурсы.

Справа:

* все выходные ресурсы.

Дополнительно:

* длительность операции;
* скорость работы устройства;
* мод-источник.

Пример:

```
Уголь + Железная руда
       ↓
      Печь
       ↓
Железный слиток
```

---

# Поддержка нескольких входов и выходов

Некоторые рецепты могут иметь:

### Много входов

Например:

```
6 ресурсов
```

или

```
9 ресурсов
```

### Много результатов

Например:

```
Шлак
Пыль
Основной продукт
Побочный продукт
```

Интерфейс должен поддерживать неограниченное количество входов и выходов.

---

# Альтернативные рецепты

Для одного предмета может существовать несколько рецептов.

Пример:

```
Железная пластина
```

можно получить через:

* молот;
* пресс;
* промышленный пресс.

При выборе цепочки пользователь должен иметь возможность переключаться между рецептами.

---

# Скорость производства

Каждая машина должна иметь производительность.

Например:

| Устройство        | Скорость |
| ----------------- | -------- |
| Печь              | 1x       |
| Электропечь       | 2x       |
| Индукционная печь | 5x       |

Эти данные используются для расчета производственных цепочек.

---

# Расчет производительности

Пользователь может указать желаемый выход.

Например:

```
100 железных пластин в минуту
```

Программа автоматически рассчитывает:

* количество машин;
* количество входных ресурсов;
* количество промежуточных продуктов;
* нагрузку на каждый этап производства.

Например:

```
100 пластин/мин
↓
4 пресса
↓
200 слитков/мин
↓
8 печей
↓
200 руды/мин
```

---

# Автоматическое построение дерева

Дополнительно пользователь может нажать:

```
Построить цепочку автоматически
```

Тогда приложение:

1. Находит рецепт предмета.
2. Рекурсивно находит рецепты ингредиентов.
3. Строит полный граф производства.

---

# Поддержка модов

Система должна быть универсальной.

Не должно быть привязки к конкретным модам.

Поддерживаются любые моды, которые добавляют:

* предметы;
* рецепты;
* машины;
* технологии.

Примеры:

* Create
* Mekanism
* Thermal Expansion
* IndustrialCraft
* Applied Energistics
* GregTech
* Immersive Engineering
* EnderIO

* Ars Nouveau

# UX-сценарий

1. Пользователь загружает моды.
2. Приложение индексирует рецепты.
3. Пользователь ищет предмет.
4. Предмет появляется на холсте.
5. Пользователь тянет связь от ингредиента.
6. Выбирает рецепт.
7. Создается узел рецепта.
8. Продолжает раскрывать ингредиенты.
9. Получает полный граф производства.
10. При необходимости включает расчет ресурсов в минуту и количества машин.

Такой подход объединяет удобство визуального редактора (как на твоем втором скриншоте) и производственные графы Factorio/Satisfactory (как на первом), но с полной поддержкой Minecraft-модов через импорт `.jar` файлов.

Количество входов и выходов

Каждый рецепт должен хранить количество каждого ресурса на входе и выходе.

Пример:

Палки

Рецепт:

2 Доски -> 4 Палки

В графе это отображается как:

[2 Доски] -> [Верстак] -> [4 Палки]
Факелы
1 Уголь + 1 Палка -> 4 Факела

Граф:

Уголь x1
          \
           -> Верстак -> Факелы x4
          /
Палка x1
Плавка железа
1 Железная руда -> 1 Железный слиток
Железная руда x1 -> Печь -> Железный слиток x1
Отображение количеств

У каждого соединения графа должно отображаться количество ресурса.

Например:

2 ── Доски ──► Верстак
              │
              ▼
          Палки x4

или как подпись возле линии:

Доски --(2)--> Верстак
Верстак --(4)--> Палки
Нормализация рецептов

Для расчетов производительности приложение должно понимать коэффициенты рецепта.

Например:

2 Доски -> 4 Палки

эквивалентно:

1 Доска -> 2 Палки

Но в интерфейсе всегда показывается оригинальный рецепт из игры.

Расчет потребления ресурсов

Если пользователь хочет получать:

100 Палок/мин

программа автоматически рассчитывает:

100 Палок/мин

Рецепт:
2 Доски -> 4 Палки

Необходимо:
50 крафтов/мин

Потребление:
100 Досок/мин
Дробные значения

При расчетах разрешены дробные значения.

Например:

7 Палок/сек

может привести к:

3.5 Доски/сек

Визуально это можно показывать как:

3.5 доски/с
7 палок/с

или

210 досок/мин
420 палок/мин
Модель данных рецепта

Я бы сразу закладывал такую структуру:

interface RecipeIO {
    itemId: string;
    amount: number;
    chance?: number; // для побочных продуктов
}

interface Recipe {
    id: string;

    machineId: string;

    inputs: RecipeIO[];
    outputs: RecipeIO[];

    durationTicks?: number;
    durationSeconds?: number;

    energyCost?: number;
}

Тогда система сможет работать с рецептами вроде:

3 Железа + 2 Угля
        ↓
      Доменная печь
        ↓
5 Стали + 1 Шлак

и даже с очень сложными модами вроде GregTech, где один рецепт может иметь 10+ входов и несколько побочных продуктов.

Я бы вообще строил внутреннюю модель не как "предметы и связи", а как двудольный граф:

Предмет -> Рецепт -> Предмет -> Рецепт -> Предмет

То есть рецепт является отдельной сущностью (нодой), а не просто линией между предметами. Именно такой подход лучше всего подходит для твоего визуального редактора и для последующих расчетов производительности.
